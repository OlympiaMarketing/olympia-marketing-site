## 4. Supabase Schema

### 4A. `pages` Table

```sql
CREATE TABLE pages (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  title TEXT NOT NULL,
  slug TEXT NOT NULL UNIQUE,
  content TEXT,                          -- HTML content from WordPress
  excerpt TEXT,
  page_type TEXT NOT NULL DEFAULT 'utility',
    -- ENUM: 'core', 'service_digital', 'service_website', 'service_traditional',
    --       'industry', 'enrollment', 'research', 'location', 'form',
    --       'scheduling', 'support', 'client_showcase', 'utility'
  parent_slug TEXT,                      -- for hierarchical pages (e.g., hosting-and-maintenance -> websites)
  seo_title TEXT,
  seo_description TEXT,
  focus_keyword TEXT,
  featured_image TEXT,                   -- URL to image
  status TEXT NOT NULL DEFAULT 'published',  -- 'published', 'draft', 'archived'
  sort_order INT DEFAULT 0,             -- for menu/nav ordering
  show_related_posts BOOLEAN DEFAULT FALSE,
  related_categories TEXT[],            -- category slugs for pulling related blog posts
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_pages_slug ON pages(slug);
CREATE INDEX idx_pages_page_type ON pages(page_type);
CREATE INDEX idx_pages_status ON pages(status);
```

### 4B. `categories` Table

```sql
CREATE TABLE categories (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,
  slug TEXT NOT NULL UNIQUE,
  description TEXT,
  parent_id UUID REFERENCES categories(id),
  sort_order INT DEFAULT 0,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_categories_slug ON categories(slug);
```

### 4C. `posts` Table

```sql
CREATE TABLE posts (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  title TEXT NOT NULL,
  slug TEXT NOT NULL UNIQUE,
  content TEXT,                          -- HTML content
  excerpt TEXT,
  seo_title TEXT,
  seo_description TEXT,
  focus_keyword TEXT,
  featured_image TEXT,
  status TEXT NOT NULL DEFAULT 'published',  -- 'published', 'draft', 'archived'
  is_featured BOOLEAN DEFAULT FALSE,    -- for homepage or section highlights
  published_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_posts_slug ON posts(slug);
CREATE INDEX idx_posts_status ON posts(status);
CREATE INDEX idx_posts_published_at ON posts(published_at DESC);
```

### 4D. `post_categories` Junction Table

```sql
CREATE TABLE post_categories (
  post_id UUID REFERENCES posts(id) ON DELETE CASCADE,
  category_id UUID REFERENCES categories(id) ON DELETE CASCADE,
  PRIMARY KEY (post_id, category_id)
);

CREATE INDEX idx_post_categories_category ON post_categories(category_id);
CREATE INDEX idx_post_categories_post ON post_categories(post_id);
```

### 4E. `page_categories` Junction Table (for related posts on pages)

```sql
-- Alternative to the TEXT[] approach on pages — a proper junction table
-- for mapping which categories show on which pages
CREATE TABLE page_categories (
  page_id UUID REFERENCES pages(id) ON DELETE CASCADE,
  category_id UUID REFERENCES categories(id) ON DELETE CASCADE,
  is_primary BOOLEAN DEFAULT TRUE,      -- primary vs secondary relevance
  PRIMARY KEY (page_id, category_id)
);

CREATE INDEX idx_page_categories_page ON page_categories(page_id);
CREATE INDEX idx_page_categories_category ON page_categories(category_id);
```

### 4F. Helper Views

```sql
-- View: Posts with their categories (for easy querying)
CREATE VIEW posts_with_categories AS
SELECT
  p.*,
  ARRAY_AGG(c.slug) AS category_slugs,
  ARRAY_AGG(c.name) AS category_names
FROM posts p
LEFT JOIN post_categories pc ON p.id = pc.post_id
LEFT JOIN categories c ON pc.category_id = c.id
GROUP BY p.id;

-- View: Pages with their related categories
CREATE VIEW pages_with_categories AS
SELECT
  pg.*,
  ARRAY_AGG(c.slug) FILTER (WHERE c.slug IS NOT NULL) AS related_category_slugs
FROM pages pg
LEFT JOIN page_categories pgc ON pg.id = pgc.page_id
LEFT JOIN categories c ON pgc.category_id = c.id
GROUP BY pg.id;
```

### 4G. Row Level Security (RLS)

```sql
-- Public read access for published content
ALTER TABLE pages ENABLE ROW LEVEL SECURITY;
ALTER TABLE posts ENABLE ROW LEVEL SECURITY;
ALTER TABLE categories ENABLE ROW LEVEL SECURITY;
ALTER TABLE post_categories ENABLE ROW LEVEL SECURITY;
ALTER TABLE page_categories ENABLE ROW LEVEL SECURITY;

-- Public can read published pages
CREATE POLICY "Public can read published pages"
  ON pages FOR SELECT
  USING (status = 'published');

-- Public can read published posts
CREATE POLICY "Public can read published posts"
  ON posts FOR SELECT
  USING (status = 'published');

-- Public can read all categories
CREATE POLICY "Public can read categories"
  ON categories FOR SELECT
  USING (true);

-- Public can read junction tables
CREATE POLICY "Public can read post_categories"
  ON post_categories FOR SELECT
  USING (true);

CREATE POLICY "Public can read page_categories"
  ON page_categories FOR SELECT
  USING (true);
```

### 5B. Dynamic Catch-All Route for CMS Pages

Since all 94 WordPress pages need to keep their exact slug as the URL, use a dynamic route at the root level:

```
app/
└── [slug]/
    └── page.tsx                                # Catches all top-level page slugs
```

This `[slug]/page.tsx` will:
1. Look up the slug in the `pages` table
2. Render the appropriate template based on `page_type`
3. Return 404 if slug not found
4. Use `generateStaticParams()` to pre-render all pages at build time

**Template mapping by `page_type`:**

| `page_type` | Template | Description |
|-------------|----------|-------------|
| `core` | `CorePageTemplate` | Full-width, custom layout per page |
| `service_digital` | `ServicePageTemplate` | Service description + CTA + related posts |

| `service_traditional` | `ServicePageTemplate` | Same template, print/PR variant |

### 5C. Blog Routes

```
app/
└── blog/
    ├── page.tsx                    # /blog — paginated index with category filter
    ├── [slug]/
    │   └── page.tsx               # /blog/[slug] — individual post
    └── category/
        └── [category]/
            └── page.tsx           # /blog/category/[category] — filtered by category
```

**Blog page features:**
- `/blog` — Shows all posts, paginated (12 per page), with category filter chips
- `/blog/[slug]` — Full post with sidebar showing related posts from same categories
- `/blog/category/[category]` — Filtered view by category slug

### 5D. Route Priority / Conflict Resolution

Since `[slug]` is a catch-all at root level, explicit routes take precedence:

```
/blog          → app/blog/page.tsx (explicit wins)
/contact       → app/contact/page.tsx (explicit wins)
/med-spa       → app/[slug]/page.tsx (dynamic catch)
/seo           → app/[slug]/page.tsx (dynamic catch)
```

Only create explicit routes for pages that need custom layouts not achievable via templates (home, blog, contact, about, work, industries, research hub).

### 5E. URL Preservation Strategy

All 94 WordPress page slugs must remain identical. The following require attention:

**Pages with potential conflicts:**
- `blog` — Has both a page AND is a route group. The explicit `/blog/page.tsx` handles this.
- `seo` — Both a page slug and a category slug. No conflict since categories are under `/blog/category/seo`.
- `support` — Page slug; explicit route recommended since it needs custom layout.

**Redirect map (if WordPress had nested URLs):**
WordPress likely served these as `olympiamarketing.com/slug` at the root. All slugs in the data are single-segment, so no nesting issues.

### 5F. `generateStaticParams` and ISR

```typescript
// app/[slug]/page.tsx
export async function generateStaticParams() {
  const { data: pages } = await supabase
    .from('pages')
    .select('slug')
    .eq('status', 'published');
  return pages.map((page) => ({ slug: page.slug }));
}

// app/blog/[slug]/page.tsx
export async function generateStaticParams() {
  const { data: posts } = await supabase
    .from('posts')
    .select('slug')
    .eq('status', 'published');
  return posts.map((post) => ({ slug: post.slug }));
}
```

Use ISR with `revalidate: 3600` (1 hour) for all CMS-driven pages.

### 6K. Implementation Pattern

```typescript
// lib/queries/getRelatedPosts.ts
export async function getRelatedPosts(pageSlug: string, limit = 6) {
  // 1. Get the page's related categories
  const { data: page } = await supabase
    .from('pages_with_categories')
    .select('related_category_slugs')
    .eq('slug', pageSlug)
    .single();

  if (!page?.related_category_slugs?.length) return [];

  // 2. Get category IDs
  const { data: categories } = await supabase
    .from('categories')
    .select('id')
    .in('slug', page.related_category_slugs);

  const categoryIds = categories.map(c => c.id);

  // 3. Get posts in those categories
  const { data: postIds } = await supabase
    .from('post_categories')
    .select('post_id')
    .in('category_id', categoryIds);

  const uniquePostIds = [...new Set(postIds.map(p => p.post_id))];

  // 4. Fetch the posts
  const { data: posts } = await supabase
    .from('posts')
    .select('id, title, slug, excerpt, featured_image, published_at')
    .in('id', uniquePostIds)
    .eq('status', 'published')
    .order('published_at', { ascending: false })
    .limit(limit);

  return posts;
}
```
